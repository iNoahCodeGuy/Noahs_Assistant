# Case study: the July 5, 2026 database outage

A production incident, a same-day rebuild, and the three latent bugs the rebuild
surfaced. Kept as a worked example of how this system fails, how failures get
detected, and what changed so this class of failure can't be silent again.

## What happened

On the morning of 2026-07-05, a routine live probe of the production chat returned
fallback answers instead of grounded ones. Nothing had alerted. The site was up,
the API returned 200s, the pipeline "worked" — but retrieval returned zero chunks
for every query, so every knowledge-base answer silently degraded to the generic
fallback, and capture writes (leads, confessions) were failing silently too.

The database — a Supabase free-tier project — had stopped resolving: NXDOMAIN on
every resolver. The dashboard showed the project as paused with corrupted metadata
(it claimed a pause date months before the project had last served traffic, and
the restore path errored). The likely cause was the free tier's inactivity
auto-pause; the corrupted state meant restore-in-place was blocked.

## The decision

Two options: wait on a support ticket to un-corrupt the pause state, or rebuild on
a fresh project from the migration files and re-embed the knowledge base from
source CSVs. The second option was chosen because the system was designed to make
it cheap: migrations in version control, the KB authored as CSVs in the repo, an
idempotent embedding pipeline, and a parity checker that could prove the rebuild
was faithful. Historical analytics rows stayed on the old project (support ticket
filed); everything the live product needs was rebuilt the same day.

## What the rebuild surfaced

Rebuilding from zero is a test almost no system runs on itself. This one failed it
three times, instructively — all three bugs had been latent in the original
deployment because it was built incrementally, never from scratch.

**1. The vector index was trained on an empty table.** Migration 001 created an
IVFFLAT index. IVFFLAT trains its centroids at build time — build it on an empty
table and the centroids are degenerate, which silently produces near-random
retrieval once data loads. The original project never hit this because its index
happened to be created after data existed. The fix: HNSW, which has no training
step and is correct in any build order. The migration file was amended, so the
next fresh build cannot repeat it.

**2. A migration that could not be applied.** Migration 003 altered a column type
that a view depended on — Postgres refuses that outright. It had "worked" in the
original database only because of the order things happened to be created there.
The file now drops and recreates the dependent view around the type change:
rebuildable from zero, in order, every time.

**3. A silent embedding-model mismatch.** The local `.env` carried a stale
`EMBEDDING_MODEL=text-embedding-ada-002` while the knowledge base was embedded
with `text-embedding-3-small`. Both models produce 1536-dimension vectors, so
nothing errors — queries just score meaningless similarities against the KB and
retrieval quietly returns garbage. The migrator now reads the same settings knob
as the retriever, so the KB and queries cannot be embedded with different models
again.

## How "recovered" was verified

Not by the site loading. The parity checker
([scripts/verify_kb_parity.py](../scripts/verify_kb_parity.py)) compares row
counts and per-chunk content hashes between the live table and the source CSVs —
it reported CLEAN on all 234 chunks. The RPC was verified to self-match at
similarity 1.0, live topical queries scored ~0.7, and the production site was
probed end-to-end: grounded answers, capture flows, dashboard.

## What changed so this can't be silent again

- **A scheduled probe.** A GitHub Actions workflow
  ([.github/workflows/keepalive.yml](../.github/workflows/keepalive.yml)) runs
  twice a week: it hits `GET /health` (which queries the KB table and returns 503
  when the database is unreachable) and asks the production chat a real
  knowledge-base question, failing loudly unless the answer is grounded. The
  traffic itself prevents the free-tier inactivity pause; the assertion turns a
  silent degradation into a red workflow run.
- **Migrations that build from zero.** Both amended files are now proven against
  a fresh database, and [supabase/migrations/APPLIED.md](../supabase/migrations/APPLIED.md)
  records exactly what ran where.
- **Provenance on every chunk.** The embedding pipeline stamps a content hash per
  chunk, so KB drift between repo and production is detectable any time, not just
  during incidents.

## The takeaways

1. **A 200 is not health.** The API masked the failure gracefully — which is
   correct for users and useless for detection. Health has to be asserted on the
   *content* of answers, not the status code.
2. **Rebuild-from-zero is a test.** Incremental systems accumulate order
   dependencies invisibly. If the disaster-recovery path is never exercised, it
   doesn't exist.
3. **Same-dimension config mismatches are the worst kind.** The embedding bug
   produced no error anywhere — only wrong similarities. Config consumed by two
   components must have one source of truth.
